## 5. 前后端 / SQL / 文档漂移

### 5.1 【高】能力资产「双轨」模型（核心架构漂移）

当前并存两套「组合/能力」存储与 API：

| 轨道 | 存储 | 主要 API | 前端消费 |
| --- | --- | --- | --- |
| **Legacy** | `tool_definition`，`kind='SKILL'` | `CompositionController` `/api/compositions`、`/api/skills` | `composition.ts`；`DomainAssignmentBoard.vue` `listCompositions`；**死页面** `CompositionList.vue` |
| **能力内核（新）** | `capability_module`、`tool_asset`、`composition_definition`、`interaction_*` | `CapabilityKernelController` `/api/capabilities/*`；`CapabilityRuntimeController` `/api/runtime/*` | `CapabilityKernel.vue`、`capabilityKernel.ts` |

**后端证据**：

- `CompositionController.java` L40–41：「底层仍是 `tool_definition`（kind=SKILL，legacy）」
- `CapabilityKernelController.java` + `CompositionDefinitionEntity` `@TableName("composition_definition")`
- `AgentFactory.java` L127：`definition.tools` 与 `definition.skills` 合并白名单，解析 **`tool_definition.id`**

**运行时证据**：

- Agent 编排 / LangGraph4j 仍围绕 **`tool_definition` 名称** 与 `AgentDefinition.skills`（`AgentDefinitionEntity.skillsJson`）
- `CompositionRuntimeExecutor` 执行 **`composition_definition` 图**（`CapabilityRuntimeController`）

**风险**：在能力内核新建的组合 **不会自动** 进入 Agent 的 `skills` 白名单（除非另有同步逻辑）；两套数据 **无单源真相**。

**建议（需产品拍板）**：

1. 明确「权威目录」：`tool_definition` vs `capability_*` 表；或
2. 做单向同步/统一 qualifiedName 空间；或
3. 文档写清「内核=模块内编排实验台，Agent=全局 tool_definition」边界。
