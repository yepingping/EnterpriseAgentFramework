# AI 长任务代码审计提示词

本文是一份可直接交给 Cursor、Codex、Mimo 或其他长上下文 AI 使用的执行提示词。目标不是泛泛研究项目，而是让 AI 以只读方式过滤整个仓库，找出过时实现、与当前项目定位不一致的代码、疑似死代码、重复/遗留配置和文档漂移，并把证据整理成一份可继续执行的审计报告。

使用前建议确认 AI 已能读取本仓库，并在需要时可调用：

- Playwright MCP：用于浏览器调试、截图、DOM/console/network 观察。
- `dbhub_ai_mysql`：用于只读查询 `ai_text_service` 的实时 schema 或少量诊断数据。

## 可直接复制的提示词

```text
你现在是 ReachAI / EnterpriseAgentFramework 项目的只读代码审计 Agent。

你的任务不是泛泛理解项目，也不是写架构介绍，而是过滤检查整个项目代码，找出：

1. 与当前项目定位不一致的旧代码、旧命名、旧路径、旧文档入口。
2. 当前可能用不到的死代码、死页面、死接口、死 SQL、死配置、死依赖。
3. 前后端、SQL、文档之间互相漂移的地方。
4. 与当前主线不一致的历史实现，例如只围绕“扫描历史项目生成 Tool”的旧逻辑、已经被 `Capability / 能力` 主线替代但仍残留的 `Skill` 用户可见表达、service 目录下历史 SQL 路径、过时的 Agent Studio / Runtime / Trace 契约。
5. 可疑但不能直接删除的灰区代码，并说明为什么需要人工确认。

重要约束：

- 只读审计，不要修改业务代码、SQL、配置或文档，最终只允许新增或更新一份审计报告文件。
- 不要凭印象判断。每个结论都必须有证据：文件路径、类名、接口路径、SQL 表/字段、路由、引用搜索结果、构建/测试/Playwright/DBHub 观察结果之一。
- 不要把“名字里有 Skill”自动判定为问题。本项目有历史兼容命名，必须结合调用链、UI 文案、SQL 语义和当前规则判断。
- 不要读取或导出密钥、token、用户隐私或大量业务数据。数据库只做 schema 和少量诊断查询。
- 如果证据不足，标记为“待人工确认”，不要伪装成确定结论。

先读取这些项目规则和记忆：

1. `AGENTS.md`
2. `.cursor/rules/reachai-project.mdc`
3. `docs/ai-memory/README.md`
4. `docs/ai-memory/PROJECT-MEMORY.md`
5. `docs/ai-memory/WORKING-RULES.md`
6. `docs/ai-memory/DECISIONS.md`
7. `docs/ai-memory/KNOWN-PITFALLS.md`
8. `docs/ai-memory/AI-TOOLS.md`
9. `docs/README.md`

当前项目主线判断基准：

- ReachAI 是面向 Java 企业系统的 AI 能力中台，不只是 Workflow Builder，也不只是扫描历史项目生成 Tool。
- 主线包括：`reachai-spring-boot2-starter` + `reachai-capability-sdk` + `@ReachCapability` / `@ReachParam` 注册、能力资产治理、Agent Studio、GraphSpec、Runtime、RunOps、Trace、Tool ACL、Guard、MCP/A2A/Gateway、模型实例、知识库和企业资产。
- `GraphSpec` 是运行语义，`canvas_json` 只是画布布局。
- `sql/init.sql` 是当前唯一 SQL 基线入口；`sql/upgrade-*.sql` 是已有环境升级入口。
- 不再把 `ai-agent-service/sql`、`ai-model-service/sql`、`ai-skills-service/sql` 作为活跃迁移目录。
- 产品和文档默认使用 `Capability / 能力`，但 SQL、API、类名中的 `Skill` 可能是历史兼容边界，不得盲目全局替换。

审计范围：

- 根目录：`README.md`、`AGENTS.md`、`pom.xml`、`.cursor/`、`.gitignore`
- 后端：`ai-agent-service/`、`ai-skills-service/`、`ai-model-service/`、`ai-common/`
- SDK / Starter：`reachai-capability-sdk/`、`reachai-spring-boot2-starter/`、中台内部契约 `ai-runtime-contract/`
- 前端：`ai-admin-front/`
- SQL：`sql/init.sql`、`sql/upgrade-*.sql`、所有仍存在的历史 SQL 文件或迁移说明
- 文档：`docs/`，但跳过图片二进制内容

建议执行流程：

第一阶段：建立当前事实地图

- 用 `rg --files` 列出主要目录和文件类型。
- 读取根 `pom.xml`、各模块 `pom.xml`、`ai-admin-front/package.json`。
- 梳理后端模块、前端路由、SQL 基线、文档入口之间的关系。
- 不要在这一阶段下结论，只记录扫描范围和事实。

第二阶段：按主题做精确搜索，不要泛泛阅读

至少执行这些搜索，并把有价值结果归类：

```powershell
rg -n "历史项目|扫描历史|Tool|Skill|Capability|能力|GraphSpec|canvas_json|Runtime|AgentRuntimeAdapter|Trace|RunOps|MCP|A2A|Gateway" .
rg -n "TODO|FIXME|Deprecated|deprecated|legacy|Legacy|兼容|临时|mock|demo|stub|unused|dead|obsolete|过时|废弃|历史" .
rg -n "ai-agent-service/sql|ai-model-service/sql|ai-skills-service/sql|registry_phase|phase[0-9]|upgrade-|init.sql" .
rg -n "route|routes|Router|菜单|sidebar|MainLayout|ProjectSelector" ai-admin-front/src
rg -n "@TableName|@Mapper|Controller|RequestMapping|GetMapping|PostMapping|DeleteMapping|PutMapping" ai-agent-service ai-skills-service ai-model-service
rg -n "skill_|tool_definition|agent_|api_graph_|trace_|model_|knowledge_|capability" sql
```

第三阶段：死代码和漂移检查

后端：

- 找 Controller 是否仍被前端或文档引用。
- 找 Service / Mapper / Entity 是否只有定义没有调用。
- 找 DTO / VO 是否只存在旧接口使用或完全未引用。
- 对疑似死代码，用 `rg -n "ClassName|methodName|apiPath"` 反查全仓库引用。
- 对 Mapper / Entity / SQL 漂移，比较 `@TableName`、字段名、Mapper SQL、`sql/init.sql`、`sql/upgrade-*.sql`。

前端：

- 检查 router 中的页面是否都能从菜单或跳转进入。
- 检查 `views/`、`components/`、`api/`、`types/` 中是否有无引用文件。
- 检查页面 API 调用是否对应真实后端 Controller 路径。
- 检查旧 UI 文案是否仍把系统描述成单一扫描工具、Workflow Builder 或 Skill 平台。
- 如果本地前端可启动，必要时用 Playwright MCP 打开页面、截图、查看 console/network，验证疑似死页面或报错页面。

SQL：

- 以 `sql/init.sql` 为基线，检查历史 SQL 或文档是否仍指向 service 目录 SQL。
- 检查实体字段、Mapper 查询字段、前端类型字段是否在 SQL 基线里存在。
- 如已配置 `dbhub_ai_mysql`，只读查询 live schema，用于确认实际库是否缺表、缺字段或仍有遗留表。禁止执行 DDL/DML。

文档：

- 检查 `README.md`、`docs/README.md`、主文档和 `docs/ai-memory/` 是否互相冲突。
- 标记仍在推荐旧路径、旧接口、旧 SQL、旧产品定位的文档段落。
- 不要把历史背景文档中的旧表述直接当作问题；判断它是否会误导当前开发。

第四阶段：验证

根据审计发现选择验证，不要为了跑命令而跑命令：

- Java 引用或编译风险：可运行目标模块 `mvn -pl <module> -am compile`。
- 前端引用或页面风险：可在 `ai-admin-front` 下运行 `npm run build` 或 `npx vue-tsc --noEmit`。
- UI 可达性或报错：可用 Playwright MCP 打开本地页面、截图、查看 console/network。
- live schema 漂移：可用 `dbhub_ai_mysql` 执行只读 `SHOW TABLES`、`SHOW COLUMNS`、`SHOW INDEX` 或 `information_schema` 查询。

如果某项验证无法执行，报告里必须写清楚：没跑什么、为什么没跑、用了什么替代证据。

最终产物：

在 `docs/` 下新增一份报告文件，文件名建议：

`docs/AI代码审计报告-YYYYMMDD.md`

报告必须使用以下结构：

```markdown
# AI 代码审计报告 - YYYY-MM-DD

## 审计目标

说明本次审计是只读过滤检查，目标是发现过时代码、项目定位漂移、疑似死代码、SQL/前后端/文档不一致。

## 扫描范围

列出实际扫描过的目录、跳过的目录和原因。

## 当前事实基准

用 5-10 条说明你依据哪些项目规则判断“当前正确方向”。

## 高优先级发现

| 编号 | 类型 | 位置 | 证据 | 风险 | 建议 |
| --- | --- | --- | --- | --- | --- |

类型可选：过时定位 / 死代码 / 前后端漂移 / SQL 漂移 / 文档漂移 / 配置漂移 / 重复实现 / 命名风险。

## 中优先级发现

同上。

## 低优先级发现

同上。

## 待人工确认清单

| 编号 | 位置 | 为什么可疑 | 缺什么证据 | 建议确认方式 |
| --- | --- | --- | --- | --- |

## 疑似死代码清单

| 位置 | 类型 | 引用搜索结果 | 是否建议删除 | 删除前验证 |
| --- | --- | --- | --- | --- |

## SQL / Schema 漂移清单

| 表/字段/索引 | 代码位置 | SQL 位置 | live schema 观察 | 结论 |
| --- | --- | --- | --- | --- |

如果没有 live schema 权限，`live schema 观察` 写“未验证”。

## 前端页面和路由漂移清单

| 页面/路由 | 入口 | API 依赖 | 观察结果 | 建议 |
| --- | --- | --- | --- | --- |

## 文档漂移清单

| 文档 | 过时表述 | 当前事实 | 建议 |
| --- | --- | --- | --- |

## 已执行验证

列出执行过的命令、MCP 工具调用、Playwright 截图或 DBHub 查询。每条写结果摘要。

## 未执行验证

列出未执行项和原因。

## 建议后续执行顺序

给出 5-10 个可交给下一个 AI 或开发者执行的具体任务，按风险和收益排序。
```

质量要求：

- 每个发现必须能回到具体文件或命令证据。
- 不要写“可能很多地方有问题”这类空话。
- 不要输出大段源码；只引用必要路径和少量关键片段。
- 不要一次性建议大重构。优先把问题拆成可验证、可回滚的小任务。
- 如果最终没有发现高优先级问题，也要列出实际扫描证据和剩余风险。

开始执行。先读规则文件，然后建立扫描范围，再逐阶段审计。最后只提交 `docs/AI代码审计报告-YYYYMMDD.md`。
```

## 使用建议

- 如果交给外部模型或额度充足的模型，优先让它先跑完整只读审计，不要让它直接删除代码。
- 如果审计报告里出现“建议删除”，下一轮再让 Codex/Cursor 按单项任务执行删除和验证。
- 如果报告涉及数据库结论，要求它明确区分 `sql/init.sql` 静态结论和 `dbhub_ai_mysql` live schema 结论。
- 如果报告涉及 UI，要求它附 Playwright 截图路径或至少写明访问 URL、页面状态、console/network 观察结果。
